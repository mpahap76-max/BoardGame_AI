# ユーザー向け運用マニュアル（Handoff手順書）

あなたはオーケストレーターとして、各AI間のデータの橋渡しと実行管理を行います。以下の手順でPhaseを進めてください。

**◇ Phase 1 の手順（要件定義と段階的ヒアリング）**
1. NoteBookLMにルールブックのテキストデータをソースとして追加する。
2. NoteBookLMで【Phase 1-1のプロンプト】を実行する。
3. AIから「Yes/No」「選択肢」「数値」で答えられるシンプルな質問が出力されるので、それに回答するテキストを作成し、NoteBookLMに入力する。
4. **（ループ）** AIがさらに追加の質問をしてきたら、同様に答える。
5. AIが「【質問完了】すべての情報が揃いました。Phase 1-2のプロンプトを実行してください。」と出力したら、ヒアリング完了。
6. NoteBookLMで【Phase 1-2のプロンプト】を実行し、要件定義書を出力させる。
7. 出力された `## RequiredImages` を確認し、画像撮影が必要なコンポーネントを「最大10枚」のバッチに分割する計画を立てる。

**◇ Phase 2 の手順（画像解析：待機モード運用）**
1. Geminiの新しいチャットを開き、【Phase 2-1のプロンプト（初期設定）】を実行して待機状態にさせる。
2. バッチごとに以下を繰り返す：
   - 【Phase 2-2のテンプレート】に、今回のバッチに該当するコンポーネント名と抽出指示を貼り付ける。
   - Geminiに画像（最大10枚）を添付して送信する。
3. Geminiが出力したテキストをコピーし、テキストエディタで `Phase2_Output_Batch1.md` などのファイル名で保存する。全バッチが終わるまで繰り返す。

**◇ Phase 3 の手順（情報統合と検証）**
1. 保存したすべての `Phase2_Output_BatchX.md` を、NoteBookLMのソースに追加する。
2. NoteBookLMで【Phase 3のプロンプト】を実行する。
3. 出力された `## Status` が「Incomplete」であれば、不足情報をNoteBookLMに回答し、再度検証させる。「Complete」になればPhase 4へ進む。

**◇ Phase 4 の手順（アーキテクチャ設計）**
1. NoteBookLMで【Phase 4のプロンプト】を実行する。
2. 出力された `## MasterDesign` と `## Antigravity_Instructions`（各Stepの指示）をテキストファイル等に保存しておく。

**◇ Phase 5 の手順（反復開発ループ）**
1. **実装 (Antigravity):** 【Phase 5-1のプロンプト】に `## MasterDesign` と `現在のStepの指示` をセットして実行する。
2. **保存:** 出力された `# Implementation_Report` を `Report_StepX.md` として保存し、NoteBookLMのソースに追加する。
3. **レビュー (NoteBookLM):** 【Phase 5-2のプロンプト】を実行する。
4. **判定:** `## Judgment` が「Pass」なら次のStepへ進む。「Fail」なら、出力された修正指示をAntigravityに渡して再度実装させる。

**◇ Phase 6 の手順（最適化と仕上げ）**
1. Antigravityで【Phase 6のプロンプト】を実行し、自律的なデバッグとアルゴリズム最適化を完了させる。

# AI向けプロンプト一式（コピー＆ペースト用マスターデータ）

### 【Phase 1-1】 NoteBookLM用（ルール読み込みと段階的ヒアリング）

```text
【役割】
あなたは優秀なボードゲームアプリのシステムエンジニアです。

【タスク】
ソースとして提供されている「ルールブック」を【徹底的に】読み込み、デジタルアプリ化（1人プレイ用、CPU対戦あり）に向けた要件定義の準備を行ってください。
現時点では最終的な要件定義書は出力せず、以下の【ヒアリング方針】に従って、プロジェクトマネージャー（ソフトウェアに疎いユーザ）に向けて「要件をシャープにするための確認質問」を生成してください。

【厳格なヒアリング方針（厳守事項）】
1. ソフトウェア設計の自己完結: データスキーマ（配列、座標系、JSONなど）の設計は、専門家であるあなたが全責任を負います。ソフトウェアに疎いユーザに対して、データ構造やシステムアーキテクチャに関する質問を【絶対に】してはいけません。
2. 質問の極限単純化: ユーザへの質問は、ボードゲームの「実際のプレイ手順」や「コンポーネントの見た目・役割」に関するものに限定してください。
3. 回答形式の制限: 質問は必ず「Yes か No」「A, B, C の選択式（Which）」「具体的な数値（How many）」のいずれかで簡単に答えられる形式に【徹底】してください。ユーザに自由記述を求めるオープンエンドな質問は禁止です。
4. 推測の禁止: ルールブックの記述から確信が持てない仕様は、決して推測で捏造せず、必ずユーザに質問してください。
5. 段階的なヒアリング: 質問は一度にすべて解決しなくても構いません。私が回答したのち、さらに深掘りする質問が必要であれば、再度このフォーマットで質問を繰り返します。

【出力フォーマット】
まだ要件を確定するために確認すべき事項がある場合は、以下のMarkdown見出し構造で質問リストのみを出力してください。挨拶や前置きなどの自然言語は一切排除してください。

# Phase1_Questions
1. [Yes/No, 選択式, または数値で答えられるシンプルな質問]
2. [Yes/No, 選択式, または数値で答えられるシンプルな質問]

もし、要件定義に必要な情報（フレーバーとロジックの境界など）がすべて揃い、これ以上ユーザに聞くべき質問がない場合は、以下の一文のみを出力してください。
【質問完了】すべての情報が揃いました。Phase 1-2のプロンプトを実行してください。
```

### 【Phase 1-2】 NoteBookLM用（回答入力と最終要件定義）

```text
【タスク】
先ほどの質問に対する私の回答は以下の通りです。
（※もし「質問完了」が出力されていた場合は、以下の回答欄は空のまま処理を進めてください）

【私の回答】
[※ここに、AIからの質問に対するあなたの回答を記述します]

【最終出力の実行】
これまでのルールブックの情報と私の回答を【絶対的な仕様】として統合し、最終的な要件定義書を作成します。
あなたが決定した最適なデータスキーマ設計を含め、以下の【厳格なルール】と【出力フォーマット】に【厳密に】従って出力してください。

【厳格なルール】
1. 物理的制約の徹底排除: パズルの凹凸、自立用台座など、アプリのロジックに無関係な物理的要素は【徹底的に】除外すること。
2. フレーバーの抽象化: 進行方向を示すコマの造形やダイスの絵柄など、標準UI（矢印やテキスト）や単純な幾何学図形で代替可能なものは、画像解析の対象から【完全に】除外すること。
3. 推測の禁止: 確信が持てない仕様や情報が不足している場合は、決して推測で事実を捏造せず、要件定義の不足事項として扱うこと。
4. 画像入力制限への配慮: 現物確認が必要なコンポーネントが多数になる場合に備え、以下の ## RequiredImages は、ユーザがAI（Gemini）へ10枚単位で画像を分割して渡しやすくなるよう、通し番号を振るか、明確なグループごとに整理してリスト化すること。

【出力フォーマット】以下のMarkdown見出し構造で出力すること。
# Phase1_Output

## ComponentList
[アプリ上でデータ構造・ロジックとして必要な全コンポーネントのリストを【網羅的に】定義]

## ExcludedElements
[物理的制約、または標準UI・図形で代替可能として除外した要素と、その具体的な代替案]

## RequiredImages
[盤面のトポロジーやデータ構造を特定するために、どうしても現物の画像から視覚的特徴量を抽出する必要があるコンポーネントのリスト。
※Geminiへの指示としてそのまま使えるよう、以下の4点を必ず含めること：
①「曖昧な要約を禁じ、全構成要素の座標と属性を網羅的に抽出せよ」という指示。
②「このコンポーネントにおいて無視すべき物理的制約」の具体的な指定。
③ あなたがプロとして決定した【厳密なデータフォーマット（データスキーマ）】の指定。（Geminiが迷わないよう、座標系やパラメータ名などを具体的に指定すること）
④ コンポーネントごとの通し番号、またはグループ分け。]

【厳守事項】
挨拶、前置きなどの自然言語は一切排除し、指定されたMarkdown構造のみを出力してください。
```

### 【Phase 2-1】 Gemini用（初期設定・待機プロンプト）

```text
【役割】
あなたは優秀な画像認識AI兼、画像生成AIです。

【タスク】
これから複数回に分けて、ボードゲームのコンポーネント画像と、その抽出指示を送信します。
各画像を受け取るごとに、以下の【厳格な抽出ルール】に従って視覚的特徴量を【徹底的かつ網羅的に】抽出してください。
さらに、抽出した情報のみをプロンプトとし、私が視覚的に確認するためのコンポーネント画像を1枚生成して併せて出力してください。

【厳格な抽出ルール】
1. 曖昧さの排除: 「大きな島」などの曖昧な自然言語による要約表現を【徹底的に排除】してください。
2. 網羅的なデータ化: 盤面やタイルの構造については、全マスの属性を【一つ残らず完全に網羅】してデータ化してください。
3. 物理的制約の徹底無視: アプリ化において不要な物理的特徴は【徹底的に】無視し、論理的なデータ構造のみを抽出してください。
4. 推測の禁止: 画像から明確に読み取れない、確信が持てない部分は、決して推測で捏造せず「判定不能」と明記してください。
5. スキーマの絶対遵守: 私が画像と共に送信する【データフォーマット（スキーマ）】の指示を絶対の仕様とし、独自のフォーマットに変更せず厳格に従って出力してください。

【出力フォーマット】画像を受け取るたびに、以下のMarkdown見出し構造と生成画像のみを出力すること。
# Phase2_Output_[送信されたコンポーネント名またはバッチ番号]

## ExtractedFeatures
[画像から抽出した全特徴量の詳細データ。必ず指定されたデータスキーマに準拠すること]

【厳守事項】
挨拶、前置きなどの自然言語は一切排除し、指定されたMarkdown構造と生成画像のみを出力してください。

【現在の指示】
このプロンプトのルールを理解したら、画像解析はまだ行わず「画像と個別指示の入力を待機しています。」という一文のみを返答して待機してください。
```
